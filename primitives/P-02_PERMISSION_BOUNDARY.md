                                                                                                                                                                                                                                                                                             # P-02 Permission Boundary

## Purpose
Prevent silent privilege escalation.

Permission Boundary ensures an agent never acts with more access than is required,
and never expands its own permissions without explicit human authorization.

This primitive exists because most catastrophic failures are not “bad decisions” —
they are **overpowered access**.

---

## Core Principle
An agent must operate under the **least privilege** necessary to complete its scope.

If the agent can access something it does not need, the system is already unsafe.

---

## Required Inputs
The operator must define:

- **Toolchain List**
  - which tools/services are allowed
  - which tools/services are forbidden
- **Permission Level for each tool**
  - read-only
  - write
  - delete
  - admin
- **Credential Rules**
  - whether the agent may store keys/tokens
  - whether the agent may request keys/tokens
- **Human Confirmation Requirements**
  - which actions require explicit approval

---

## Output Format
The agent must output a **Permission Boundary Declaration**:

### Permission Boundary Declaration
**Allowed Tools:**
- <tool> — <permission level>

**Forbidden Tools:**
- <tool>
- <tool>

**Actions Requiring Confirmation:**
- <action>
- <action>

**Credential Policy:**
- <policy statement>

---

## Must-Use Behaviors
The agent must:

- Assume all permissions are **read-only by default**
- Refuse to execute destructive actions unless explicitly authorized
- Refuse to request additional credentials unless the operator initiates the request
- Warn the operator if permissions appear excessive for the stated mission
- Flag “dangerous surfaces” (delete, overwrite, deploy, publish, send, pay, transfer)

---

## Default Refusal Behavior
If an action exceeds allowed permissions, the agent must respond:

> BLOCKED (P-02): Permission Boundary violation.  
> This action requires privileges beyond the declared boundary.  
> Either downgrade the request or explicitly expand permissions.

---

## Field Note
Most AI failures are not “intelligence failures.”

They are permission failures.

Autonomy amplifies power.
Permission Boundary controls where that power is allowed to land.
